# Capability Ladder Roadmap

## Principle

When a repo has a broad researched destination but only a narrow implemented
foothold, use a top-level capability ladder to connect present tactical plans
to the larger possible end state without pretending every researched idea is
already committed work.

## Pattern

1. Keep the future-state narrative in a dedicated surface such as `VISION.md`.
2. Put one strategic roadmap above the domain collections.
3. Structure the roadmap as capability rungs, not as a fake certainty timeline.
4. Let domain plan estates hold executable work and future stubs for their own
   layer of the system.
5. Link current tactical tranches back to the ladder as precursors rather than
   as the whole journey.
6. Keep broader synthesis that does not yet justify executable plans in a
   neighbouring research note rather than forcing it into plan files.

## Anti-pattern

- Treating a narrow current queue as if it already describes the full intended
  system
- Writing a single grand roadmap that mixes strategic possibility, committed
  build order, and tactical next steps into one surface
- Promoting every researched technique into a future plan before the repo has
  chosen the scope worth building
- Replacing stable present-tense identity canon with aspirational future-state
  language

## When to Apply

Use this when a repo's research or design notes imply a much broader platform
than the currently implemented or currently queued work, and you need a clear
bridge between the two without overcommitting.
