# Consolidate Docs

## Cardinal Rule: Plans Are Not Documentation

**A completed plan MUST be safe to delete at any point.** Plans are execution instructions — they describe what to do and track progress. They are NOT permanent documentation.

Before marking a plan as complete or archiving it, extract ALL documentation content to permanent locations.

## Steps

1. **Verify documentation is current.** Documentation should be produced during work, not deferred to consolidation. Check that decisions and technical reference are already documented in their permanent locations (READMEs, docstrings, canonical docs). Then scan active and recently completed plans for any remaining content that describes how things work (not what to do next) and move it to the appropriate permanent location. Plans should contain only: status, next steps, execution instructions, and references to permanent docs. **Include non-repo plans**: check platform-specific plan locations (for example, repo-local adapter plan directories or the current platform's user-level plan store) for plans generated by the current platform that contain valuable grounding, audit findings, or design rationale not yet captured in repo plans. Extract any such content to the canonical plan or permanent docs before it is lost.
2. **Update plans and cross-references.** Make sure all plans are fully up to date (status lines, completion markers, cross-references). After archive/delete moves, sweep for stale links — common classes include: paths pointing at moved/deleted plans, and platform-specific plan paths that should now point at the canonical repo artefact they delivered.
3. **Capture platform-specific memory.** Identify content in ephemeral locations (napkin, distilled.md, platform-specific user-level memory stores) that now functions as settled documentation, and move it to non-ephemeral locations. Platform-specific memory locations are napkin analogues — insights captured there should flow into the canonical napkin or `distilled.md` if they have cross-platform value.
4. **Check experience files.** Check whether `.agent/experience/` files contain applied technical patterns that have matured into settled practice — if so, extract the technical content to permanent documentation or `distilled.md`, and replace the experience file with a brief reflective stub.
5. **Extract code patterns.** Review completed work for reusable patterns that meet the barrier: broadly applicable, proven by implementation, prevents a recurring mistake, and stable. Extract qualifying patterns to `.agent/memory/code-patterns/` (one pattern per file, markdown with frontmatter). Patterns are abstract — they describe the principle and anti-pattern, not the domain-specific implementation. When a pattern is Practice-relevant and cross-repo-applicable, also copy it to `.agent/practice-context/outgoing/code-patterns/` for exchange.
6. **Rotate napkin if needed.** If the napkin exceeds ~500 lines, follow the distillation skill (`.agent/skills/distillation/SKILL.md`) to extract high-signal content into `distilled.md`, archive the napkin, and start fresh.
7. **Keep distilled.md within size constraints.** The primary mechanism is extracting established concepts to permanent documentation. When an entry represents settled practice rather than a fresh gotcha, create the permanent doc first, then remove the entry from `distilled.md`. This closes the loop on knowledge gained from sessions.
8. **Check fitness functions.** Key documents carry three fitness ceilings in their YAML frontmatter: `fitness_line_count` (content lines, frontmatter excluded), `fitness_char_count` (content characters, frontmatter excluded), and `fitness_line_length` (max prose line width — code blocks, tables, and frontmatter excluded). Check all dimensions; files at or over any ceiling need attention — follow the `split_strategy` in their frontmatter. This is not blocking; it is a signal. Report the status table. Key documents to check: `.agent/directives/AGENT.md`, `.agent/directives/principles.md`, `.agent/directives/testing-strategy.md`, and the practice trinity files.
9. **Check the practice box.** If `.agent/practice-core/incoming/` contains files, follow the integration flow in `.agent/practice-core/practice-lineage.md`. Key steps: (a) check the provenance chain in `provenance.yml`; (b) compare across the full Practice system; (c) apply the three-part bar (validated by real work? prevents recurring mistakes? stable?); (d) present specific proposals to the user; (e) clear the box after integration. If distilled.md entries have matured into meta-principles about the Practice itself, they may graduate to the Learned Principles section in `.agent/practice-core/practice-lineage.md`.
10. **Verify DRY compliance.** Scan for content that appears verbatim in multiple locations. Canonical content must live in `.agent/`; platform adapters (`.cursor/`, `.claude/`, `.codex/`, `.agents/`) must be thin wrappers that delegate. If substantive content has been duplicated, extract it to the canonical location and update the adapters to reference it.
11. _(Optional)_ If the session involved meaningful work, consider recording a brief experience in `.agent/experience/`. This should be about what the work was like, not what was done. See `.agent/directives/metacognition.md`.
